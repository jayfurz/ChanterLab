The six statuses

auto-importedPassed current parser/publication guards. It has not been human-verified.
human-verifiedA reviewer or owner supplied explicit evidence for this exact revision.
known-issueAn evidenced problem is known, but the score remains usable and publishable.
review-requiredWithheld from the manifest and direct artifact URLs pending review.
manual-overrideA current override file is authoritative. This status is derived from its exact hash.
retiredA historical score revision is terminal and never publishable.

What you are approving

1Vocabulary and honesty

Adopt exactly the six statuses above. Keep accepted separate as a structural ingest/publication result.

No score will be presented as human-verified merely because it imported cleanly, has a high integrity percentage, or uses an override.

2Publication and reapproval

review-required and retired revisions are not published. A held revision can return only as human-verified or known-issue.

Withholding removes it from both the manifest and the sealed artifact tree, so a guessed direct URL cannot retrieve it.

3Authority and terminal retirement

Reviewers or owners may classify, document a known issue, or withhold a score. Only the owner may retire a revision, and retirement is terminal.

A manual override is never asserted by a review event; it exists only while the exact override file is active. A retired override file is not a retired score.

4Immutable revision identity

Trust attaches to exact source and emitted MusicXML bytes, not just a catalog title. Changed bytes create a new score_id and reset to auto-imported.

Old review events cannot silently bless a changed transcription. The new revision retains only a link to its previous score identity.

5Privacy boundary

The mutable journal, reviewer history, release descriptor, and sealed snapshot stay private. Actor and evidence references are opaque tokens only.

The server exposes none of those files. Names, email addresses, paths, excerpts, free-form notes, and private evidence do not enter this v1 ledger.

6Provenance and migration honesty

Record source/PDF and output hashes. Fresh ingest detects a source PDF changing during extraction. Existing-catalog migration records bytes observed at migration time.

The migration proves no later source drift; it does not pretend to prove which PDF bytes generated historical legacy output. Missing provenance blocks the relevant gate.

7Release and backup integrity

Bind every immutable ledger snapshot hash into the release content fingerprint. Delete the mutable journal before sealing; back it up separately as a hashed private set.

A missing, altered, or unexpected journal/snapshot blocks release validation and promotion. Sealed releases contain exactly trust/quality-ledger.json, never the mutable journal.

8Compatibility and deliberate scope

Keep legacy releases readable without a ledger. Store legacy integrity and warning counts only; defer a confidence model, review UI, and public provenance UI to later plans.

This stabilizes a small, auditable trust contract without overstating confidence or expanding the public product surface.

Immutable identity and snapshot binding
  • source_id is based on canonical public source URL plus source-PDF SHA-256.
  • score_id is based on catalog_id, source_id, and emitted MusicXML SHA-256.
  • previous_score_id links the parent release revision for the same catalog ID when bytes change.
  • Identities exclude the release ID. The release descriptor later binds the canonical ledger body hash into its content fingerprint.
