# samples/ — the "Voices" instrument's audio assets

GitHub issue #66: add a sampled instrument voice behind a "Sound: Synth /
Voices" toggle, alongside the existing Tone.PolySynth triangle-wave synth.
This directory holds everything backing "Voices" mode: `voices/*.ogg` (what
the app actually fetches, lazily, only when Voices is selected) plus the two
scripts used to generate them.

## Provenance — why generated, not downloaded

The issue's priority order asked for real CC0/public-domain samples first,
generation only as the fallback ("when in doubt, generate"). Two CC0 sources
were actually checked (not just recalled from memory):

- **VSCO-2 Community Edition** — <https://github.com/sgossner/VSCO-2-CE>.
  `LICENSE` at the repo root is the verbatim **CC0 1.0 Universal** text
  (confirmed by fetching
  `https://raw.githubusercontent.com/sgossner/VSCO-2-CE/master/LICENSE`).
  Contents: Brass, Keys (incl. a real pipe organ — `Keys/Organ/Loud/
  Rode_Man3Open_*.wav`, conveniently already multisampled every 3 semitones),
  Percussion, Strings, Woodwinds. **No human voice/choir content at all.**
- **VCSL (Versilian Community Sample Library)** —
  <https://github.com/sgossner/VCSL>, explicitly "CC0 general-purpose ...
  broader expansion to the VSCO 2 CE sample set" (its own README). Organized
  by Hornbostel-Sachs class (Aerophones/Chordophones/Electrophones/
  Idiophones/Membranophones). Human voice would fall under "Free
  Aerophones," which contains only harmonicas and a siren — **no vocal
  samples either.**

No other CC0 multi-note vocal "ah"/choir set was found that could be
fetched *and* have its license verified in the time available. Freesound-
style single-file libraries mix CC0/CC-BY/CC-BY-NC per upload and don't
offer a coherent multisampled instrument from one consistently-licensed
source, so per the issue's explicit instruction — reject unclear licensing,
generate instead — that's the path taken here.

**Decision:** `generate-voices.mjs` synthesizes an original "ah"-vowel pad
offline (additive harmonic synthesis through a static formant filter bank —
see that file's header for the DSP detail: 4 formant peaks tuned to /ɑ/,
slow-fade-in vibrato, pitch jitter, onset breathiness, a musical decay tail).
**Zero external audio assets are used anywhere in this pipeline** — every
sample byte is generated by that script's own math. There is therefore no
third-party copyright or licensing question to track: this audio is
original work product of this project, same as the surrounding JS.

The pipe organ option (real, CC0, verified above) was considered and set
aside for this issue: the toggle is literally labeled "Voices," and an organ
patch would misrepresent what the button says it does. It remains a
legitimate, ready-to-use CC0 source if a future issue wants a second/third
instrument choice.

## How the files were made

```
node samples/generate-voices.mjs   # -> samples/voices/ah_<midi>.wav (11 notes)
./samples/encode-voices.sh         # -> samples/voices/ah_<midi>.ogg, deletes the .wav
```

- **Notes:** MIDI 40, 44, 48, 52, 56, 60, 64, 68, 72, 76, 80 (E2 up to
  ~G#5/A5's neighborhood, every 4 semitones) — covers the app's ~E2-A5 SATB
  span with Tone.Sampler's own pitch-shifting filling in the ≤2-semitone
  gaps between samples.
- **Format:** mono, 44.1kHz, ~3.5s render, encoded to Ogg/Opus at 64kbps
  (see `encode-voices.sh`'s header for why 3.5s and why the specific
  bitrate/headroom numbers).
- **Total shipped payload:** 11 files, ~328KB combined (`du -cb
  samples/voices/*.ogg`) — comfortably under the issue's ~3MB budget, with
  room to spare for a second instrument choice later without threatening it.
- **Format choice (Ogg/Opus over .wav/.mp3):** Chromium (this app's smoke-
  test/CI browser) and every real target browser decode Ogg Opus natively
  via the same `fetch()` + `AudioContext.decodeAudioData()` path
  Tone.Sampler already uses for any URL — verified against this app's own
  vendored `vendor/Tone.js` build (Tone.Sampler / Tone.ToneAudioBuffers just
  hand the fetched bytes to `decodeAudioData`; the codec is the browser's
  problem, not Tone's). At the tiny sizes here, `.mp3` would cost roughly
  2-3x the bytes for the same perceived quality on this kind of sustained
  pad material — no reason to pay for it.
